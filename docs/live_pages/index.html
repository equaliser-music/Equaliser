Artist Onboarding

An anonymous applicant submits a request via /join, an operator approves it, and the artist redeems the resulting invite code. Backup save + profile setup come before the dashboard — the artist's Kind 0 is discovered from Damus / Primal / nos.lol / relay.nostr.band if present, or created fresh if not.

Label Onboarding

An operator generates a label invite code directly. The new label redeems, saves a backup, then goes through the same profile-setup step as artists do — no exceptions — before landing on the label dashboard.

Artist Joins a Label Roster

An existing label calls "Add Existing Artist", picks the relationship type (Phase G — Managed or Signed), generates the roster invite code, shares it out-of-band, and the artist redeems with their own keys (managed_by = label, relationship_type set per the picker). The artist then goes through a profile create/update flow before reaching the dashboard.

Architecture overview

Every flow ends with a row in node_artists (or node_operators for the very first operator). That row determines the user's role and what they can manage. The relevant API endpoints:

  • POST /api/access/request — public submission from /join form
  • POST /api/label/access-requests/{id}/approve — operator/label approves, generates code
  • POST /api/label/invite-codes — direct invite-code generation (operator picks role)
  • POST /api/label/add-existing-artist — labels generate roster invite codes
  • POST /api/access/redeem — atomic claim: marks code used + inserts node_artists row

After redemption, every artist and label is routed through /admin/profile-setup.html before reaching the dashboard. The page queries Damus, Primal, nos.lol and relay.nostr.band for an existing Kind 0 and pre-fills the form when found; otherwise it presents a blank "create profile" form. Kind 0 + Kind 10002 (NIP-65 relay list) are published to the local relay (always) and any external relays the user selects. There is no shortcut — Kind 0 must be on the local relay before the dashboard is reachable.

For multi-label nodes, every artist's identity is held by the artist (non-custodial). The label can request a NIP-26 delegation to publish on the artist's behalf — see the delegation plan.
